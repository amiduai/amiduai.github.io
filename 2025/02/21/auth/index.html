认证漏洞

从概念上讲，认证漏洞很容易理解。然而，由于认证与安全之间存在着明确的关联，这些漏洞通常都至关重要。

认证漏洞可能会让攻击者获取敏感数据并使用相关功能。它们还会暴露出更多的攻击面，便于攻击者进一步利用。因此，学习如何识别和利用认证漏洞，以及如何绕过常见的防护措施，是非常重要的。

在本节中，我们将阐述：

  • 网站常用的最常见认证机制。
  • 这些机制中潜在的漏洞。
  • 不同认证机制中固有的漏洞。
  • 因实施不当而产生的典型漏洞。
  • 如何使自己的认证机制尽可能强大。

什么是认证？

认证是验证用户或客户端身份的过程。网站对任何连接到互联网的人都可能是开放的。这使得强大的认证机制成为有效网络安全不可或缺的一部分。

认证主要有三种类型：

  1. 你知道什么，例如密码或安全问题的答案。这些有时被称为 “知识因素”。
  2. 你拥有什么，这是一个物理对象，例如手机或安全令牌。这些有时被称为 “占有因素”。
  3. 你本身或你的行为特征。例如，你的生物特征或行为模式。这些有时被称为 “固有因素”。
    认证机制依赖一系列技术来验证上述一个或多个因素。

认证和授权有什么区别？

认证是验证用户是否为其声称身份的过程。授权则涉及验证用户是否被允许执行某项操作。
例如，认证决定了试图使用用户名 “Carlos123” 访问网站的人是否真的是创建该账户的那个人。

一旦 “Carlos123” 通过了认证，其权限将决定他们被授权执行的操作。例如，他们可能被授权访问其他用户的个人信息，或者执行诸如删除其他用户账户之类的操作。

认证漏洞是如何产生的？

认证机制中的大多数漏洞以以下两种方式之一出现：

  • 认证机制很薄弱，因为它们无法充分抵御暴力破解攻击。
  • 实施过程中的逻辑缺陷或糟糕的编码使得攻击者能够完全绕过认证机制。这有时被称为 “失效的认证”。

在网络开发的许多领域中，逻辑缺陷会导致网站出现意外行为，这可能是也可能不是安全问题。然而，由于认证对安全至关重要，有缺陷的认证逻辑很可能会使网站面临安全问题。

存在漏洞的认证会有什么影响？

认证漏洞的影响可能非常严重。如果攻击者绕过认证或通过暴力破解进入另一个用户的账户，他们就可以访问该被攻陷账户拥有的所有数据和功能。如果他们能够攻陷一个高权限账户，例如系统管理员账户，他们就可以完全控制整个应用程序，并有可能访问内部基础设施。

即使攻陷一个低权限账户，攻击者也可能获取到他们本不应拥有的数据，例如具有商业敏感性的业务信息。即使该账户无法访问任何敏感数据，它仍然可能使攻击者访问更多页面，从而提供了更多的攻击面。通常，从公开可访问的页面发起高严重性的攻击是不可能的，但从内部页面则有可能做到。

认证机制中的漏洞

网站的认证系统通常由几个不同的机制组成，漏洞可能在这些机制中出现。有些漏洞适用于所有这些情况。而其他漏洞则更特定于所提供的功能。

我们将更深入地研究以下几个方面中一些最常见的漏洞：

  • 基于密码登录的漏洞（认证漏洞 二）
  • 多因素认证的漏洞（认证漏洞 三）
  • 其他认证机制的漏洞 （认证漏洞 四）

如何保护你的认证机制

接下来，我们将讨论如何防止我们所讨论的一些漏洞出现在你的认证机制中。

认证是一个复杂的话题，而且正如我们所展示的那样，很不幸的是，弱点和缺陷很容易不知不觉地出现。显然，要列出你可以采取的每一项保护自己网站的措施是不可能的。然而，有几个通用原则是你应该始终遵循的。

妥善处理用户凭证

如果你在不经意间向攻击者泄露了一组有效的登录凭证，那么即使是最强大的认证机制也会失效。不言而喻，你绝不应通过未加密的连接发送任何登录数据。尽管你可能已经为登录请求实施了 HTTPS 协议，但也要确保通过将任何尝试的 HTTP 请求重定向到 HTTPS 来强制使用该协议。
你还应该对网站进行审查，以确保不会通过公开可访问的个人资料或在 HTTP 响应中泄露任何用户名或电子邮件地址。

不要依赖用户来保障安全

严格的认证措施通常需要用户付出一些额外的努力。由于人性使然，一些用户几乎不可避免地会想办法省去这些麻烦。因此，你需要尽可能地强制推行安全行为。

最明显的例子是实施有效的密码策略。一些较为传统的密码策略之所以失败，是因为人们会想方设法使用自己能想到的、可预测的密码来符合这些策略。相反，实施某种简单的密码检查器可能会更有效，它可以让用户尝试设置密码，并实时反馈密码的强度。一个广为人知的例子是由 Dropbox 开发的 JavaScript 库 zxcvbn。通过只允许使用密码检查器评估为高强度的密码，你可以比使用传统策略更有效地强制使用安全密码。

防止用户名枚举

如果你透露系统中存在某个用户，那么攻击者攻破你的认证机制就会容易得多。甚至在某些情况下，由于网站的性质，知道某个人拥有一个账户这件事本身就属于敏感信息。

无论尝试登录的用户名是否有效，使用相同的、通用的错误消息都很重要，并且要确保这些消息确实完全相同。每次登录请求都应返回相同的 HTTP 状态码，最后，要让不同场景下的响应时间尽可能难以区分。

实施强大的暴力破解防护措施

鉴于构造一次暴力破解攻击是多么容易，采取措施防止或至少干扰任何暴力破解登录的尝试就至关重要。

一种比较有效的方法是实施严格的、基于 IP 地址的用户速率限制。这应该包括防止攻击者操纵其表面 IP 地址的措施。理想情况下，在达到一定限制后，你应该要求用户每次登录尝试都完成验证码（CAPTCHA）测试。

请记住，这并不能保证完全消除暴力破解的威胁。然而，让暴力破解过程尽可能繁琐且需要手动操作，会增加潜在攻击者放弃并转而寻找更易攻击目标的可能性。

反复检查你的验证逻辑

正如我们的实验所展示的那样，简单的逻辑缺陷很容易潜入代码中，而在认证相关的代码中，这些缺陷有可能彻底危及你的网站和用户。彻底审查任何验证或确认逻辑以消除缺陷，是实现强大认证的绝对关键。一个可以被绕过的检查，最终与根本没有检查相比，也好不到哪里去。

不要忘记辅助功能

确保不要只关注核心登录页面，而忽略与认证相关的其他功能。当攻击者可以自由注册自己的账户并探索这些功能时，这一点尤为重要。请记住，密码重置或修改与主登录机制一样，都是有效的攻击面，因此，其安全性也必须同样强大。

实施恰当的多因素认证

虽然多因素认证可能并不适用于每个网站，但如果实施得当，它比仅基于密码的登录要安全得多。请记住，验证同一因素的多个实例并不是真正的多因素认证。通过电子邮件发送验证码本质上只是一种更繁琐的单因素认证形式。

基于短信的双因素认证（2FA）在技术上验证了两个因素（你知道的东西和你拥有的东西）。然而，例如，通过 SIM 卡更换进行滥用的可能性意味着这种系统可能不可靠。

理想情况下，双因素认证应该使用专门的设备或应用程序来直接生成验证码。由于这些设备和应用程序是专门为提供安全保障而设计的，所以通常更安全。

最后，就像主认证逻辑一样，要确保你的双因素认证检查中的逻辑是合理的，这样才不会轻易被绕过。
